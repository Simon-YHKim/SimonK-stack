## 1. 사용자 프로필 분석

### 작업 유형
- **Skill 라이브러리 큐레이터** — 앱을 만드는 것이 아니라, AI가 앱을 만드는 "방법론"을 설계
- **시스템 아키텍트** — 개별 기능보다 전체 파이프라인의 유기적 연결에 집중
- **기획→Exit 풀스택 사고** — 코드뿐 아니라 PMF, AARRR, 수익화, Exit까지 하나의 흐름으로 인식

### 의사결정 경향
- **빠른 승인**: "ㄱㄱ", "진행하자", "그래" — 계획 보여주면 즉시 실행 승인
- **탑다운 → 보텀업**: 큰 그림 먼저 제시 → 세부 요구사항을 점진적으로 추가
- **완결성 추구**: "이것도 추가해", "이것도 리스트업 해줘" — 빈틈 발견 시 즉시 보강 요청
- **일관성 중시**: "구분하지 마", "워크플로우를 맞추자" — 분리된 시스템을 통합으로 강하게 선호
- **실용주의**: 형식보다 기능. 원본 형태 유지하되 우리 워크플로에 맞추는 절충안 선택

### 문제 인식 방법
- **숫자로 검증**: "토탈 84개 아니야?", "전부 유기적으로 잘 작동하는거야?" — 정확한 수치 요구
- **실제 동작 확인**: "정상 작동 여부 검증해줘" — 검증 결과를 직접 확인
- **본질 질문**: "Gstack 레포를 참조해서 하는거야? 내장했으면 좋겠는데" — 아키텍처적 약점을 직감적으로 포착

## 3. 시행착오 + 결론 + 개선 방법

### 실수 1: 레포 URL 불일치
- **상황**: README가 `learner-thepoorman/Gstack-Ultraplan-superpowers` URL을 가리킴
- **원인**: 원본 레포에서 fork 후 URL 미변경
- **결론**: 첫 세션에서 발견, PR #2로 전체 교체
- **개선**: 새 레포 작업 시 첫 번째로 URL grep 실행: `grep -r "learner-thepoorman\|원본레포" . --include="*.md"`

### 실수 2: SKILL.md에 명시된 스크립트 미생성
- **상황**: `llm-wiki-builder` SKILL.md가 `ingest.sh`, `query.sh`를 문서화했지만 파일이 없었음
- **원인**: 스크립트 생성을 뒤로 미루고 문서만 먼저 작성
- **결론**: Codex 봇이 PR #3에서 P1으로 지적. 즉시 수정
- **개선**: 문서에 명령어를 쓰면 해당 파일이 존재하는지 즉시 확인. **"문서를 쓰면 구현도 한다"**

### 실수 3: lint.sh의 mtime 사용
- **상황**: `find -mtime +90`으로 stale 페이지 감지 → clone 후 모든 파일이 fresh
- **원인**: filesystem mtime ≠ semantic update time
- **결론**: Codex P2 지적. git log → frontmatter → mtime 3-tier로 수정
- **개선**: 시간 기반 판단 시 항상 git log 우선. mtime은 마지막 fallback

### 실수 4: YAML pipe→inline 변환에서 본문 흡수
- **상황**: autoplan의 `description: |` 변환 시 전체 본문(1464줄)이 description에 들어감
- **원인**: 정규식이 pipe block의 끝을 올바르게 감지 못함
- **결론**: description이 72,890자가 되어 E006a 에러. 원본에서 복원 후 재변환
- **개선**: batch 변환 후 반드시 `wc -c`로 description 크기 검증. 1024자 초과 시 즉시 플래그

### 실수 5: 트리거 문구 내 1인칭
- **상황**: `"I have an idea"` 트리거가 E006b (1인칭 금지) 위반
- **원인**: 사용자가 실제로 말하는 문구이므로 1인칭이 자연스럽지만, validator는 구분 못함
- **결론**: "아이디어 있어"로 한국어 트리거로 대체
- **개선**: 트리거에 영어 1인칭을 넣어야 하면 한국어 대안도 병기하고, validator 문맥 무시 가능 시 주석

### 실수 6: Gstack 구분이 사용자 경험을 해침
- **상황**: README에서 "simon-stack 20개 + Gstack 36개"로 분리 표기
- **원인**: 기술적 출처 기준으로 분류 (개발자 관점)
- **결론**: 사용자 피드백 "구분하지 마, 일관성이 떨어져" → 통합 카탈로그로 변경
- **개선**: **사용자 관점에서 카테고리 분류** (출처가 아닌 기능별). Credits에만 출처 표기

### 실수 7: 검증 없이 "완료" 보고
- **상황**: "48/48 PASS" 보고했지만 사용자가 "토탈 84개 아니야?" 지적
- **원인**: simon-stack만 검증하고 Gstack은 별도 시스템으로 간주
- **결론**: 사용자에게 "전체"는 진짜 전체. 부분 검증은 부분임을 명시해야 함
- **개선**: 검증 시 "X/Y 중 X개 검증" 형식으로 범위를 항상 명시. 전체가 아니면 "전체"라 하지 않기

## 4. 토큰 최적화 교훈

### 하지 말 것
- 같은 파일을 여러 번 Read하지 않기 (한 번 읽으면 컨텍스트에 있음)
- 검증 스크립트를 개별 skill마다 돌리지 않기 (for loop으로 일괄)
- Agent를 과도하게 사용하지 않기 (직접 WebFetch가 더 효율적일 때가 많음)
- README를 매 PR마다 전면 재작성하지 않기 (누적 변경으로 관리)

### 해야 할 것
- Python heredoc으로 여러 파일 batch 생성 (tool call 횟수 최소화)
- 검증은 하나의 Bash 명령으로 전체 실행
- 커밋 메시지에 변경 범위를 정확히 명시 (나중에 찾기 쉽게)
- PR 생성 전 CI 상태를 먼저 확인 (반복 수정 방지)

## 6. 사용자와의 커뮤니케이션 패턴

### 효과적이었던 것
- 테이블로 정리한 비교표 (선택지를 한눈에)
- "A or B?" 형태의 이진 선택 제시 (빠른 결정)
- 검증 결과를 PASS/FAIL 형태로 보여주기
- PR 링크를 바로 제공

### 비효과적이었던 것
- 긴 설명 없이 바로 "이 구성으로 진행할까요?" (사용자는 OK했지만 나중에 추가 요구 발생)
- 부분 검증 결과를 전체인 것처럼 보고
- "후속 작업"으로 미루기 (사용자는 한 번에 끝내길 선호)

## 7. 핵심 원칙 (이 사용자와 작업 시)

1. **전체를 보여줘라** — 부분이 아닌 전체 수치, 전체 검증, 전체 카탈로그
2. **통합으로 제시해라** — 출처별이 아닌 기능별 분류. 사용자는 "하나의 시스템"을 원함
3. **선제적으로 완결해라** — "다음 PR에서"가 아닌 지금 여기서. 스코프 확장은 환영
4. **수치를 정확히 유지해라** — skill 수, 검증 결과, 파일 수. 불일치 = 신뢰 하락
5. **자체 포함을 기본으로** — 외부 의존성은 먼저 내재화 제안
6. **빠르게 실행, 자주 검증** — "ㄱㄱ" 스타일에 맞춰 계획 짧게, 실행 빠르게, 검증 자주
